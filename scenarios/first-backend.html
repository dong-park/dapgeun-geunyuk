Backend Engineer 첫 번째 링크

Backend Engineer로 구독 신청했을 때 첫 번째로 보내는 학습페이지입니다. 질문은 단순합니다. “재고 차감 요청은 큐로 직렬화하고 별도의 락은 쓰지 않겠습니다.” 이 말을 면접에서 안전하게 방어하려면, 먼저 락을 쓰는 선택지들이 무엇을 해결하고 무엇을 포기하는지까지 설명해야 합니다.

참조 관점: Martin Fowler의 Optimistic/Pessimistic Offline Lock, PostgreSQL Transaction Isolation/Application-level consistency, Redis Streams consumer group 모델, lost update 사례 기반 설명.

1. 먼저 사고를 터뜨려보자

상황

한정판 키트 재고가 딱 1개 남았습니다. A와 B가 거의 동시에 결제 버튼을 눌렀습니다. 서비스 코드는 아래처럼 단순합니다.

재고 조회 → 재고가 0보다 크면 주문 생성 → 재고 -1 저장

여기서 위험한 부분은 “트랜잭션을 썼는가”보다 두 요청이 같은 재고 값을 보고 같은 결정을 할 수 있는가입니다.

예측 질문. 아래 흐름에서 가장 먼저 의심해야 할 문제는?
  1. 트랜잭션을 쓰면 항상 안전하니 문제없다.
  2. A와 B가 모두 재고 1을 보고 주문 성공 판단을 할 수 있다.
  3. 재고가 1이면 DB가 자동으로 한 명만 성공시킨다.
정답 보기

정답은 2번입니다. 위험은 “둘 다 같은 재고 1을 읽었다”에서 시작합니다. 이후 저장 순서에 따라 lost update, 초과 판매, 결제 후 품절 같은 문제가 생길 수 있습니다.

2. 실패 흐름을 타임라인으로 보면 바로 보인다

시간 A 요청 B 요청 DB 재고
T1 재고 조회: 1 1
T2 재고 조회: 1 1
T3 “재고 있음” 판단, 주문 생성 “재고 있음” 판단, 주문 생성 1
T4 재고 0 저장 재고 0 저장 0
겉으로는 정상처럼 보이지만

DB 재고는 0입니다. 그런데 성공 주문은 2건입니다. 이게 더 위험합니다. 재고가 -1이면 이상 징후라도 보이지만, 이 경우에는 숫자만 보면 정상처럼 보일 수 있습니다.

3. “트랜잭션 썼습니다”만으로는 답이 부족하다

트랜잭션은 작업 묶음의 원자성을 보장하는 데 중요합니다. 하지만 기본 격리 수준에서 두 트랜잭션이 같은 값을 읽고 각자 판단하는 상황까지 항상 자동으로 막아주지는 않습니다. 그래서 좋은 답변은 “트랜잭션을 씁니다”에서 멈추지 않고, 먼저 지켜야 할 불변식을 말합니다.

재고 차감의 핵심 불변식

같은 상품에 대해 성공 주문 수는 실제 재고 수를 넘으면 안 됩니다. 그리고 같은 주문 요청은 여러 번 전달되어도 한 번만 재고에 반영되어야 합니다.

4. 선택지 A: 비관적 락

비관적 락은 “충돌이 날 것”이라고 보고 먼저 잠급니다. 예를 들어 재고 row를 SELECT ... FOR UPDATE로 잡고, 한 트랜잭션이 재고를 확인하고 차감하는 동안 다른 트랜잭션은 기다리게 합니다.

좋은 점

이해하기 쉽고, 같은 row를 동시에 고치는 일을 강하게 막습니다. 충돌이 자주 나고 잘못되면 안 되는 데이터에는 안전한 선택지가 될 수 있습니다.

불편한 점

사람이 몰리면 요청이 줄을 섭니다. 트랜잭션이 길어지면 DB 커넥션과 락을 오래 붙잡습니다. 여러 테이블이나 여러 상품을 다른 순서로 잠그면 데드락도 고려해야 합니다.

그래서 비관적 락은 “항상 더 안전한 정답”이 아니라, 정확성은 강하지만 대기와 처리량 비용을 내는 방식입니다.

5. 선택지 B: 낙관적 락

낙관적 락은 먼저 막지 않습니다. 대신 저장 시점에 “내가 읽은 버전이 아직 유효한가”를 확인합니다. 충돌이 드문 상황이라면 잠금 비용을 줄이고, 충돌이 난 요청만 실패 또는 재시도하게 할 수 있습니다.

update stock set quantity = quantity - 1, version = version + 1

where product_id = ? and version = ? and quantity > 0

업데이트된 row 수가 0이면 누군가 먼저 바꿨거나 재고가 없다는 뜻입니다. 이때 재시도하거나 품절로 응답합니다. 다만 이벤트 상품처럼 10만 명이 동시에 몰리면 낙관적 락은 재시도 폭풍을 만들 수 있습니다. 실패한 요청들이 다시 읽고 다시 쓰려고 몰리기 때문입니다.

6. 선택지 C: 조건부 업데이트

단순 재고 차감만 보면 더 직접적인 방법도 있습니다. “재고가 있을 때만 차감한다”를 하나의 원자적 업데이트로 표현하는 것입니다.

update stock set quantity = quantity - 1

where product_id = ? and quantity > 0

성공 row 수가 1이면 차감 성공, 0이면 품절입니다. 이 방식은 단순하고 강력합니다. 하지만 실제 주문 시스템에서는 주문 생성, 결제 승인, 재고 차감, 실패 보상, 알림이 함께 움직입니다. 그래서 조건부 업데이트만으로 “전체 주문 흐름”이 끝나는 것은 아닙니다. 주문 상태 모델이 필요합니다.

7. 선택지 D: 큐 기반 직렬화

큐 기반 접근은 DB row를 동시에 두드리게 하지 않고, 같은 상품의 재고 차감 작업을 순서대로 처리하는 방식입니다. Redis Streams의 consumer group이나 일반 메시지 큐를 떠올리면 됩니다. 핵심은 “락을 안 쓴다”가 아니라 처리 흐름을 직렬화해서 충돌 지점을 줄인다입니다.

면접에서 쓸 수 있는 핵심 문장

재고 차감 요청은 큐를 통해 단일 처리 흐름으로 직렬화하여 동시성 문제를 완화하고, 별도의 락 메커니즘은 사용하지 않는 방향으로 설계하겠습니다.

하지만 이 문장만 말하면 부족합니다. 큐는 새로운 문제를 만듭니다. 메시지가 중복 전달될 수 있고, 워커가 처리 중 죽을 수 있고, 큐가 밀리면 사용자가 결과를 기다려야 합니다. 그래서 큐 기반 설계는 반드시 아래 장치와 함께 말해야 합니다.

멱등키 주문 상태 상품 단위 파티셔닝 재시도/보상 큐 지연 모니터링

예를 들어 같은 requestId는 한 번만 차감되게 하고, 주문은 처음부터 “성공”이 아니라 접수됨 → 재고 확인 중 → 확정/품절 상태를 갖게 합니다. 같은 상품의 메시지는 같은 파티션으로 보내 순서를 지키고, 다른 상품은 병렬 처리합니다.

8. 상황별로 선택이 달라진다

상황 추천 방향 이유
이벤트 상품 재고 조건부 업데이트 + 큐/파티셔닝 고려 동일 상품에 요청이 집중되고 초과 판매 비용이 큽니다.
선착순 쿠폰 Redis 원자 연산 + DB 유니크 제약 즉시 성공/실패 UX가 중요하고 중복 발급을 막아야 합니다.
포인트 차감 강한 트랜잭션 + 멱등성 + 감사 로그 금전성 데이터라 정합성과 추적 가능성이 중요합니다.
프로필 수정 낙관적 락 충돌이 드물고 실패해도 사용자가 다시 저장할 수 있습니다.
좋아요 수 비동기 집계/원자 증가 정확한 즉시성보다 처리량과 UX가 더 중요할 수 있습니다.

9. 흔한 오개념 정리

오개념 1. 트랜잭션만 쓰면 재고 동시성은 자동으로 해결된다

트랜잭션은 필요하지만 충분조건은 아닙니다. 어떤 격리 수준인지, 읽은 값으로 판단한 뒤 쓰는지, 조건부 업데이트인지, row lock을 잡는지에 따라 결과가 달라집니다.

오개념 2. 비관적 락은 안전하니 항상 쓰면 된다

안전성은 좋아질 수 있지만 대기, 처리량 저하, 데드락, 장애 전파 비용이 생깁니다. 충돌 비용이 낮은 데이터까지 모두 강하게 잠그면 시스템이 불필요하게 무거워집니다.

오개념 3. 큐를 쓰면 동시성 문제는 끝난다

큐는 처리 순서를 만들지만 메시지 중복, 워커 실패, 지연, 보상 처리 문제가 남습니다. 큐 기반 설계의 핵심은 멱등성과 상태 관리입니다.

10. 판단형 퀴즈

Q1. 이벤트 상품에 낙관적 락만 적용했을 때 생길 수 있는 대표 문제는?
  1. 락 대기 때문에 모든 요청이 DB에서 줄을 선다.
  2. 버전 컬럼이 있으면 재시도는 절대 발생하지 않는다.
  3. 충돌 요청이 계속 실패하고 재시도하면서 트래픽이 더 커질 수 있다.
정답 보기

정답은 3번입니다. 낙관적 락은 충돌이 드문 곳에 잘 맞습니다. 충돌이 극단적으로 많은 이벤트에서는 실패와 재시도가 폭증할 수 있습니다.

Q2. 큐 기반 재고 차감에서 가장 먼저 같이 설계해야 하는 것은?
  1. 같은 요청이 두 번 들어와도 한 번만 반영되게 하는 멱등성
  2. 모든 상품을 하나의 워커에서 처리하는 전역 순서
  3. 사용자에게 무조건 즉시 주문 성공을 보여주는 화면
정답 보기

정답은 1번입니다. 큐는 at-least-once 전달일 수 있고, 워커 실패 후 재처리될 수 있습니다. 같은 요청이 중복 반영되지 않도록 멱등키가 필요합니다.

12. 면접 답변 빌더

아쉬운 답변

“재고는 중요하니까 비관적 락을 쓰겠습니다.”

문제: 왜 그 방식인지, 비용은 무엇인지, 큐나 조건부 업데이트와 비교가 없습니다.

괜찮은 답변

“재고는 초과 차감되면 안 되므로 동시성 제어가 필요합니다. 충돌이 많다면 비관적 락이나 조건부 업데이트를 고려하고, 충돌이 적으면 낙관적 락도 사용할 수 있습니다.”

괜찮지만 아직 실제 주문 흐름, UX, 실패 복구가 약합니다.

강한 답변

“재고 차감은 초과 판매가 발생하면 바로 비즈니스 손실로 이어지는 민감한 영역입니다. 먼저 불변식은 성공 주문 수가 실제 재고를 넘지 않는 것입니다. 단순 차감은 quantity > 0 조건부 업데이트로 원자성을 확보할 수 있고, 이벤트처럼 동일 상품에 요청이 몰리는 상황에서는 같은 상품의 차감 요청을 큐로 보내 단일 처리 흐름으로 직렬화하는 방식을 고려하겠습니다. 이 경우 별도의 락 메커니즘은 최소화할 수 있지만, 큐 지연과 중복 메시지, 워커 실패가 생길 수 있으므로 멱등키, 주문 상태, 재시도/보상 정책, 큐 지연 모니터링을 함께 설계하겠습니다.”

13. 참고한 데이터/레퍼런스

  • Martin Fowler — Optimistic Offline Lock / Pessimistic Offline Lock: 충돌을 먼저 막을지, 커밋 시점에 감지할지의 패턴 의도.
  • PostgreSQL Documentation — Transaction Isolation: 트랜잭션과 격리 수준이 애플리케이션 불변식을 자동으로 모두 보장하지 않는다는 점.
  • Redis Streams Documentation — Consumer groups: 큐/컨슈머 그룹 기반 직렬 처리와 at-least-once 처리 모델.
  • Lost update 사례: “둘 다 같은 값을 읽고 각자 저장했는데 최종 숫자는 정상처럼 보이는” 재고 장애를 설명하기 위한 대표 패턴.
