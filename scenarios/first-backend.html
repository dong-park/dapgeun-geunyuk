재고는 민감한데, 락 없이 큐로 처리한다면?

SoftwareBackend중급12분

Backend Engineer 첫 번째 링크

Backend Engineer로 구독 신청했을 때 첫 번째로 보내는 학습페이지입니다. 재고 차감처럼 정합성이 중요한 데이터를 큐 기반 직렬화로 처리한다고 말할 때, 어디까지 설명해야 안전한 답변인지 연습합니다.

1. 실제 상황

커머스 서비스에서 한정 수량 상품 1개가 남았습니다. 주문 요청은 동시에 여러 개 들어올 수 있고, 팀은 “재고 차감 요청은 큐를 통해 단일 처리 흐름으로 직렬화하고, 별도의 락 메커니즘은 사용하지 않겠다”고 설계 방향을 잡았습니다. 면접관은 묻습니다. “락이 없는데 overselling을 어떻게 막나요? 큐가 죽거나 중복 메시지가 오면요?”

질문: 이 설계를 면접에서 어떻게 방어적으로 설명하겠습니까?

4. 답변 비교

약한 답변

큐에 넣으면 순서대로 처리되니까 동시성 문제는 없어집니다. 그래서 락은 안 걸어도 됩니다.

괜찮은 답변

재고 차감 요청을 큐에 넣고 단일 worker가 순차 처리하면 같은 상품에 대한 동시 차감을 줄일 수 있습니다. 다만 worker를 여러 개 두면 상품 단위 순서가 깨질 수 있으니 같은 상품은 같은 파티션으로 보내야 합니다.

강한 답변

재고가 민감하므로 요청을 바로 DB에서 경쟁시키기보다 상품 단위로 큐에 직렬화합니다. 핵심은 “큐를 쓴다”가 아니라 같은 stock_id의 메시지가 같은 파티션/단일 consumer 흐름으로 처리된다는 보장입니다. 그래도 큐는 중복 전달과 재시도가 가능하므로 order_id/request_id 기반 멱등 처리, conditional update 같은 DB 최종 방어선, 실패 메시지 DLQ, 큐 lag와 차감 실패율 모니터링을 함께 둡니다. 그래서 별도 분산 락은 도입하지 않되, 직렬화 경계와 멱등성, 원자적 차감 조건으로 정합성을 방어하겠습니다.

5. 면접관 꼬리질문

  1. consumer를 2개 이상 늘리면 어떤 문제가 생기나요?
  2. 큐는 at-least-once인데 중복 차감은 어떻게 막나요?
  3. 조건부 업데이트도 쓸 거면 락을 안 쓴다는 말은 정확히 어떤 의미인가요?
  4. 큐 lag가 30초까지 늘어나면 사용자 경험은 어떻게 설계하나요?
