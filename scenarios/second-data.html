대시보드 숫자와 결제 DB 숫자가 다르다

DataMetric Quality중급12분

Data Analyst / Scientist 두 번째 링크

Data Analyst / Scientist의 두 번째 학습페이지입니다. 숫자 차이를 단순 SQL 오류가 아니라 정의, 파이프라인, 지연, 중복의 문제로 진단하는 연습입니다.

1. 실제 상황

월 매출 리포트에서 BI 대시보드는 12.4억을 보여주는데, 결제 DB 집계는 11.8억입니다. 재무팀은 오늘 숫자를 확정해야 하고, PM은 BI 숫자를 기준으로 보고서를 만들었습니다. ETL은 하루 4번 돌고, 환불/부분취소 이벤트는 별도 테이블에 있습니다.

질문: 어떤 순서로 원인을 좁히고, 오늘 어떤 숫자를 보고 기준으로 삼겠습니까?

4. 답변 비교

약한 답변

BI 쿼리가 틀린 것 같으니 SQL을 다시 확인하겠습니다.

괜찮은 답변

먼저 매출 정의가 같은지 보고, ETL 지연이나 환불 반영 여부, 타임존 차이를 확인하겠습니다. 재무 확정은 결제 원장 기준으로 하는 게 안전합니다.

강한 답변

숫자 불일치는 SQL 하나의 문제가 아니라 정의와 파이프라인 문제일 수 있습니다. 먼저 매출 정의가 gross인지 net인지, 환불/부분취소/쿠폰/세금 포함 여부가 같은지 맞춥니다. 그 다음 ETL freshness, 중복 결제 이벤트, 타임존, late-arriving data를 확인합니다. 오늘 확정 숫자는 source of truth인 결제 원장 기준으로 두고, BI는 차이 원인과 보정 여부를 표시하겠습니다. 재발 방지를 위해 metric definition 문서, freshness 알림, reconciliation test를 추가하겠습니다.

5. 면접관 꼬리질문

  1. source of truth는 항상 production DB인가요?
  2. late-arriving data가 있으면 대시보드는 어떻게 보여줘야 하나요?
  3. 재무팀과 PM이 원하는 숫자 정의가 다르면?
