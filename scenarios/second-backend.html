1. 면접관의 추가 상황

1라운드에서 큐 기반 직렬화로 재고 차감을 설명했습니다. 이제 면접관이 상황을 바꿉니다. 주문 메시지를 consumer가 받아 DB 재고를 차감한 직후, “처리 완료 ack”를 보내기 전에 worker가 죽었습니다. 큐는 같은 메시지를 다시 전달했고, 사용자는 같은 주문이 두 번 처리될까 봐 위험한 상태입니다.

2라운드 질문: 이 상황에서 중복 차감과 유실을 어떻게 막겠습니까?

4. 답변 비교

약한 답변

worker가 죽으면 큐가 다시 처리하면 되니까 괜찮습니다.

괜찮은 답변

큐는 중복 전달될 수 있으므로 order_id 기준으로 이미 처리한 요청인지 확인하고, 처리 완료 후 ack를 보내겠습니다.

강한 답변

이 장애는 at-least-once 큐에서 흔한 “처리는 됐는데 ack 전 죽음” 케이스입니다. 재전달은 정상 동작이므로 consumer는 반드시 멱등해야 합니다. order_id나 reservation_id를 unique key로 둔 처리 이력/재고 예약 테이블을 만들고, 같은 key는 기존 결과를 반환하게 합니다. 재고 차감도 `stock >= qty` 조건부 업데이트와 처리 이력 저장을 같은 트랜잭션으로 묶어 중복 차감을 막겠습니다. 실패 메시지는 DLQ로 보내고, 재처리 성공률/중복 탐지율/큐 lag를 모니터링하겠습니다.
