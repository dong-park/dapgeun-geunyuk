Backend Engineer 두 번째 링크

Backend Engineer의 두 번째 학습페이지입니다. 성능 개선 수단인 캐시가 장애 상황에서 어떻게 stampede, stale data, invalidation 문제를 만드는지 설명하는 연습입니다.

1. 실제 상황

상품 상세 API가 느려져 Redis 캐시를 붙였습니다. 평소에는 응답 시간이 800ms에서 80ms로 줄었습니다. 그런데 인기 상품 세일 시작 직후 캐시 TTL이 동시에 만료되며 DB QPS가 20배로 튀고, 일부 사용자는 품절 상품을 구매 가능한 상태로 봤습니다. 면접관은 묻습니다. “캐시를 붙였는데 왜 더 위험해졌고, 어떻게 설계하겠습니까?”

질문: 캐시 성능 개선과 데이터 정합성 리스크를 어떻게 균형 있게 설명하겠습니까?

4. 답변 비교

약한 답변

Redis 캐시를 쓰면 DB 부하가 줄어들기 때문에 TTL을 길게 잡으면 됩니다.

괜찮은 답변

캐시 만료가 동시에 몰리면 DB로 요청이 쏠릴 수 있으니 TTL에 jitter를 주고, 인기 키는 미리 갱신하겠습니다. 재고처럼 민감한 값은 캐시에만 의존하지 않겠습니다.

강한 답변

캐시는 평균 응답 시간을 줄이지만, 만료가 몰리면 cache stampede로 DB를 더 크게 때릴 수 있습니다. 그래서 TTL jitter, stale-while-revalidate, single-flight, hot key prewarm을 조합해 miss 폭주를 줄이겠습니다. 또 상품 설명처럼 약간 stale해도 되는 데이터와 가격/재고처럼 민감한 데이터를 분리하고, 구매 가능 여부는 최종 주문 시점의 DB 조건부 검증으로 방어합니다. 지표는 hit rate뿐 아니라 miss QPS, hot key, DB fallback, stale 응답 비율까지 보겠습니다.
