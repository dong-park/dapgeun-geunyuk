모바일에서만 버튼이 두 번 눌려 결제가 중복됐다

FrontendUX Reliability중급10분

Frontend Engineer 두 번째 링크

Frontend Engineer의 두 번째 학습페이지입니다. UI 버그를 단순 disable 처리로 끝내지 않고 네트워크, idempotency, 사용자 피드백까지 연결하는 연습입니다.

1. 실제 상황

모바일 결제 페이지에서 사용자가 구매 버튼을 한 번 눌렀는데 결제가 두 번 생성됐다는 CS가 들어왔습니다. 로그를 보니 일부 기기에서 터치 이벤트가 중복 발생했고, 네트워크가 느릴 때 사용자가 다시 버튼을 누른 케이스도 있습니다. 백엔드는 “프론트에서 버튼 disable 해주세요”라고 합니다.

질문: 프론트엔드 입장에서 어디까지 책임지고, 백엔드와 어떤 계약을 맞추겠습니까?

4. 답변 비교

약한 답변

버튼을 누르면 바로 disabled 처리해서 중복 클릭을 막겠습니다.

괜찮은 답변

프론트에서는 loading 상태와 버튼 disable, 중복 submit 방지를 넣고, 백엔드에도 같은 요청을 한 번만 처리할 수 있도록 요청 id를 보내겠습니다.

강한 답변

프론트의 버튼 disable은 필요하지만 최종 방어선은 아닙니다. 터치 이벤트 중복, 네트워크 지연, 브라우저 재시도, 사용자의 새로고침까지 고려하면 결제 요청에는 idempotency key가 필요합니다. 프론트는 요청 생성 시 고유 key를 만들고 loading/disabled/중복 submit 방지를 적용하며, 실패와 처리 중 상태를 명확히 보여줍니다. 백엔드는 같은 key의 요청을 한 번만 처리하고 기존 결과를 반환해야 합니다. 이후 모바일 기기별 이벤트 로그와 중복 요청률을 계측해 재발을 봅니다.
