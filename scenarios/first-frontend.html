Frontend Engineer 첫 번째 링크

Frontend Engineer로 구독 신청했을 때 첫 번째로 보내는 학습페이지입니다. “느려요”를 렌더링/네트워크/API/데이터 크기 관점으로 분해하는 답변근육을 연습합니다.

1. 실제 상황

B2B 관리자 대시보드 첫 화면 로딩이 2초에서 8초로 늘었습니다. PM은 “프론트 최적화 좀 해주세요”라고 말합니다. 그런데 최근 변경은 차트 라이브러리 교체, API 응답 필드 추가, 권한 체크 API 추가가 모두 섞여 있습니다.

질문: 어디서부터 확인하고, 어떤 순서로 개선하겠습니까?

4. 답변 비교

약한 답변

React 렌더링이 느린 것 같아서 memo와 lazy loading을 적용하겠습니다.

괜찮은 답변

먼저 Network와 Performance 탭으로 병목이 API인지 JS 번들인지 렌더링인지 나누겠습니다. 이후 번들 분리, 차트 lazy loading, API 필드 축소 같은 개선을 적용하겠습니다.

강한 답변

먼저 “느리다”를 TTFB, API latency, JS download/parse, render, chart hydration으로 나눠 측정하겠습니다. 최근 변경이 여러 개이므로 Performance trace와 Network waterfall, 실제 사용자 RUM을 보고 병목을 특정합니다. 빠른 완화로는 above-the-fold 우선 렌더링, 차트 lazy loading/skeleton, 불필요 필드 제거를 적용하고, 구조적으로는 API 집계/캐싱, 번들 budget, 성능 회귀 알림을 둡니다. 프론트 최적화만 말하지 않고 사용자 체감 지표와 재발 방지까지 연결하겠습니다.
