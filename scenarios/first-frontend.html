Lighthouse는 좋은데 실제 사용자는 느리다

오늘 목표는 React memo나 lazy loading을 외우는 게 아닙니다. “느리다”는 피드백을 실사용자 지표, 네트워크, JS 실행, 렌더링, API 경계로 나누고 어떤 개선이 실제 체감 속도를 바꾸는지 판단하는 연습입니다.

Frontend Engineer 첫 번째 링크

Frontend Engineer에게 첫 번째로 보내는 성능 진단 페이지입니다. “프론트 최적화 해주세요”라는 요청을 그대로 받지 않고, lab metric과 field metric을 구분해 병목을 찾는 답변근육을 만듭니다.

1. 먼저 사고를 터뜨려보자

상황

B2B 관리자 대시보드 첫 화면 로딩이 2초에서 8초로 늘었습니다. PM은 “프론트 최적화 좀 해주세요”라고 말합니다. 그런데 최근 변경은 차트 라이브러리 교체, API 응답 필드 추가, 권한 체크 API 추가, 서드파티 스크립트 삽입이 모두 섞여 있습니다.

예측 질문. 가장 먼저 하면 위험한 행동은?
  1. React.memo를 전체 컴포넌트에 적용한다.
  2. Network waterfall과 Performance trace로 시간을 분해한다.
  3. 실제 사용자 RUM에서 기기/네트워크별 분포를 본다.
정답 보기

정답은 1번이 위험합니다. memo는 원인 확인 없이 쓰면 복잡도만 늘릴 수 있습니다. 먼저 TTFB, API latency, JS download/parse, rendering, hydration, third-party 비용으로 나눠야 합니다.

2. 실패 흐름을 타임라인으로 보면 바로 보인다

구간 관측 의심 원인 확인 도구
TTFB 첫 byte가 1.8초 서버/API/캐시 miss Network, server log
JS parse main bundle 1.2MB 차트 라이브러리/불필요 dependency Bundle analyzer, Performance
Render 차트 mount 2.5초 DOM 과다/동기 계산 Performance flame chart
Field 저사양 Android에서만 P75 악화 lab과 field 차이 RUM, Web Vitals

3. 좋은 답변은 “진단 기준”부터 세운다

면접에서 강한 답변은 바로 해결책을 외치지 않습니다. 먼저 어떤 결과가 위험한지, 어떤 지표나 증거로 원인을 나눌지, 그리고 어떤 선택지가 어떤 비용을 만드는지 설명합니다. 그래야 “그럴듯한 개선안”이 아니라 실제 서비스에서 쓸 수 있는 판단으로 들립니다.

4. 측정 A: Lab metric과 Field metric을 나눈다

Lighthouse는 통제된 환경에서 재현하기 좋지만 실제 사용자 환경을 대표하지 못할 수 있습니다. CrUX/RUM의 LCP, INP, TTFB, long task, device/network segment를 함께 봐야 “점수는 좋은데 고객은 느린” 상황을 설명할 수 있습니다.

5. 측정 B: Network waterfall로 API와 리소스를 분리한다

느림이 프론트 렌더링인지 API 응답인지 먼저 나눕니다. 권한 체크 API가 직렬로 붙었는지, 대시보드 API가 과도한 필드를 내려주는지, 이미지/폰트/서드파티가 critical path에 있는지 확인합니다.

6. 개선 C: 빠른 완화와 구조 개선을 구분한다

빠른 완화는 skeleton, above-the-fold 우선 렌더링, 차트 lazy loading, prefetch, 캐시입니다. 구조 개선은 API 집계/필드 축소, 번들 budget, route-level code splitting, 성능 회귀 알림입니다. 면접에서는 둘을 섞지 않고 순서로 말해야 합니다.

7. 상황별 선택이 달라진다

상황 추천 방향 이유
LCP 악화 critical resource 최적화/SSR/이미지 최적화 첫 화면 체감이 핵심입니다.
INP 악화 long task 분해/useTransition/debounce 상호작용 지연은 렌더링보다 JS 실행 문제가 많습니다.
API latency 캐시/집계 API/병렬화 프론트만 고쳐서는 한계가 있습니다.
저사양 기기 악화 bundle 축소/차트 지연 로딩 평균보다 P75/P95 사용자 경험이 중요합니다.

8. 흔한 오개념 정리

오개념 1. Lighthouse 점수만 좋으면 성능은 괜찮다

Lab 점수는 출발점일 뿐입니다. 실제 사용자는 느린 기기, 불안정한 네트워크, 로그인 상태, 캐시 miss를 겪습니다.

오개념 2. React.memo를 많이 쓰면 빨라진다

memo는 비교 비용과 복잡도를 만듭니다. re-render가 병목이라는 증거가 있을 때 써야 합니다.

오개념 3. API가 느리면 프론트는 할 일이 없다

UX 완화, 캐싱, 요청 병렬화, progressive rendering으로 체감 속도를 줄일 수 있습니다.

11. 면접 답변 빌더

아쉬운 답변

“React 렌더링이 느린 것 같아서 memo와 lazy loading을 적용하겠습니다.”

괜찮은 답변

“Network와 Performance 탭으로 API, JS 번들, 렌더링 병목을 나누고 번들 분리나 lazy loading을 적용하겠습니다.”

강한 답변

“먼저 느리다는 말을 TTFB, API latency, JS download/parse, render, interaction delay로 분해하겠습니다. Lighthouse 같은 lab metric만 보지 않고 RUM/Web Vitals에서 P75 사용자와 기기·네트워크 segment를 확인합니다. 병목이 API면 캐시/집계/병렬화, JS면 route-level splitting과 차트 lazy loading, 렌더링이면 long task 분해와 virtualization을 적용합니다. 빠른 완화로 skeleton과 above-the-fold 우선 렌더링을 두고, 재발 방지를 위해 performance budget과 회귀 알림을 운영하겠습니다.”

12. 참고한 데이터/레퍼런스

  • web.dev Web Vitals: LCP/INP/CLS와 사용자 체감 지표 기준.
  • Chrome DevTools Performance/Network documentation: waterfall, long task, flame chart 기반 병목 분석.
  • React documentation — memo/useMemo/useCallback: 최적화 도구의 비용과 사용 조건.
  • Real User Monitoring 패턴: 평균이 아니라 P75/P95와 segment로 성능을 판단하는 방식.
